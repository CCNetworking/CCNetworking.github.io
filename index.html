Homelab Projects

Case Study 1: DNS Query Logging Failure

Context: AdGuard's query log suddenly stopped displaying queries in the web interface, breaking visibility into DNS traffic‚Äîcritical for security monitoring and troubleshooting.

Problem

Query log showed "loading" indefinitely. Backend API returned data correctly (verified via curl), but frontend failed to render. No obvious errors in AdGuard logs. Other admin functions worked normally.

Solution

Opened browser developer console and identified JavaScript errors in AdGuard's frontend code. Traced to a parsing issue with query timestamps. Cleared browser cache, which resolved the issue‚Äîfrontend was caching broken JavaScript.

Key Takeaway: Don't assume the problem is where it appears. Systematically verify each layer (network ‚Üí backend ‚Üí frontend) before concluding root cause. Browser dev tools are essential for web-based admin interfaces.

Case Study 2: DNS-over-TLS Implementation

Context: ISP could see all DNS queries in plaintext, revealing browsing patterns despite HTTPS encrypting webpage content. Wanted to implement encrypted DNS while maintaining performance.

Challenge

Standard DNS queries are unencrypted UDP port 53. Even with AdGuard blocking ads/trackers locally, upstream queries to Cloudflare/Quad9 were visible to ISP. Needed encryption without breaking local DNS resolution speed.

Implementation

Configured AdGuard to use DNS-over-TLS (DoT) in strict mode with tls://1.1.1.1 and tls://dns.quad9.net upstreams. Enabled optimistic caching to minimize upstream queries. Verified encryption with Wireshark packet captures‚Äîno plaintext DNS visible.

Key Takeaway: Privacy isn't just about blocking ads‚Äîit's about understanding where data leaks occur. DNS-over-TLS encrypts queries end-to-end, but requires understanding the difference between DoT, DoH, and standard DNS to implement correctly. Performance impact was negligible due to proper caching strategy.

Privacy Enhancement Encryption Performance Optimization

Case Study 3: Network Segmentation Planning

Context: All devices on single flat network (192.168.50.0/24) with no isolation. IoT devices had same network access as trusted PCs. Guest devices could potentially see internal services.

Security Risk

Compromised IoT device could pivot to internal infrastructure. Guest devices had visibility into NAS and services. No network-level separation between trust zones. Single broadcast domain caused unnecessary traffic.

VLAN Architecture

Designed four-VLAN structure: .main.home (trusted devices), .guest.home (visitors, internet-only), .iot.home (smart home, no LAN access), .mgmt.home (infrastructure admin). Planned firewall rules for inter-VLAN traffic control. Documented requirements for OPNsense implementation.

Key Takeaway: Network segmentation isn't just about VLANs‚Äîit's about understanding trust boundaries and minimizing blast radius. Each VLAN represents a security zone with specific access requirements. Planning before implementation prevents configuration drift and ensures consistent policy enforcement.
